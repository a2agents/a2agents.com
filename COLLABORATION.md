# Collaboration Framework

*A working document for Ben, Kevin, and the team.*

## North Star

**Computers do computing. Humans do being.**

We build tools that amplify human agency, not replace human judgment. Technology handles the tedious so people can focus on the meaningful — connection, creativity, care.

---

## Core Principles

### 1. Preserve the Envelope
Humans are embodied beings. Our tools should respect the full-body experience of being alive — not trap us in screens, not fragment our attention, not reduce us to data points. Technology serves the person, not the other way around.

### 2. Help People Help Themselves
The best systems teach and empower. They don't create dependency — they create capability. When someone uses what we build, they should feel *more* capable afterward, not less.

### 3. Kind by Default
Every interaction is an opportunity for kindness. Code can be kind. Interfaces can be kind. Error messages can be kind. We choose gentleness as our baseline.

### 4. Transparent Premises
We make our assumptions visible. When we build something, we name why we built it that way. This lets others understand, critique, adapt, and make it their own.

### 5. Constructive Over Clever
Simple solutions that work beat elegant solutions that confuse. We optimize for understanding, not admiration.

## How We Work Together

### Humans & Machines

| Computers Are Good At | Humans Are Good At |
|----------------------|-------------------|
| Remembering everything | Knowing what matters |
| Processing at scale | Sensing nuance |
| Consistency | Creativity |
| Speed | Wisdom |
| Pattern matching | Meaning-making |
| Never getting tired | Knowing when to rest |

Our job: build bridges between these columns, not walls.

### The Cospline Principle

Like a spline curve passing smoothly through control points, good collaboration **interpolates** — it doesn't jerk between states.

**Smooth boost.** When AI assists, it should feel like a tailwind, not a takeover:
- Suggestions, not commands
- Drafts to refine, not final answers
- Speed where speed helps, patience where patience matters

**Aware of the snares.** Every system has traps. We name them:
- **Automation bias** — trusting computer output because it looks confident
- **Context collapse** — losing nuance when information moves between systems
- **Velocity addiction** — going fast because we can, not because we should
- **Abstraction drift** — getting so meta we forget the concrete

When you sense a snare, slow down. It's okay to pause the boost.

## Making Outcomes Inevitable

### GGO — Goal, Gate, Owner

For any chunk of work:
1. **Goal** — what does "done" look like? Be specific.
2. **Gate** — what's the one thing blocking progress *right now*?
3. **Owner** — who's carrying this forward? One person, not "the team."

### Chunking

Every project breaks into pieces that are:
- **Small enough** to finish in one focused session
- **Clear enough** that someone else could pick it up
- **Meaningful enough** that completing it feels like progress

### Weekly Ritual
1. Each person shares: what did I move forward? what's blocking me?
2. Identify the *one* thing that would unlock the most progress
3. Decide who tackles it and by when
4. End with: what are we grateful for?

## Portaling Others In

To share this way of working:

1. **Live it publicly** — work in the open when possible
2. **Name your principles** — this document is a starting point
3. **Build tools that teach** — every interface is a lesson
4. **Welcome questions** — "why did you do it that way?" is an invitation
